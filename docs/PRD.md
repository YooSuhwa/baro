### 1. 서비스 개요

- **한 줄 소개**: 경쟁사 실시간 모니터링 & 비교 플랫폼
- **핵심 문제**:
  - **As-Is**: 경쟁사 뉴스 수집, 스펙 정리, 비교표 작성을 수동으로 수행. 담당자가 주 1~2회 반나절(월 20~30시간)을 소모. 컨플루언스 등에 수동 기록하여 정보가 분산되고 최신 상태 유지가 어려움.
  - **To-Be**: 경쟁사/제품을 등록만 해두면 뉴스가 자동 수집·분류되고, 스펙 변경이 감지되어 관리자 승인 후 반영. 제품 비교 시 스마트폰 비교 사이트처럼 나란히 비교 가능.
- **핵심 가치**: 자동 수집, 팩트 기반 비교, 의사결정 가속

### 2. 사용자 정의

| 사용자 유형 | 설명 | 주요 행동 |
|------------|------|----------|
| 관리자 (Admin) | 경쟁사/제품 등록, 스펙 승인을 담당하는 실무자 | 경쟁사·제품 CRUD, 비교 항목 템플릿 관리, 스펙 변경 승인/거절, 수동 수집 트리거 |
| 뷰어 (Viewer) | 수집된 정보를 조회·비교하는 사용자 (경영진, 기획팀, 제품팀) | 대시보드 조회, 뉴스 피드 열람, 제품 비교표 조회, 뉴스 동향 비교 |

**권한 정책 (MVP)**:
- MVP에서는 인증 없이 모든 사용자가 접근 가능 (사내 네트워크 전제)
- 단, **관리 기능**(경쟁사 등록, 스펙 승인 등)은 URL 경로 `/admin/*`으로 분리하여 접근 제한
- 추후 v1.x에서 이메일+비밀번호 자체 회원가입 도입 시 역할(Admin/Viewer) 기반 권한 제어로 전환

| P1 | 스펙 변경 감지·승인 | AI가 스펙 변경을 감지하여 관리자에게 제안 → 승인/거절 워크플로 | Admin |

**Not Doing (MVP에서 하지 않을 것)**:
- SNS(X, LinkedIn) 크롤링 — API 비용 및 정책 리스크
- 자동 보고서 생성 — MVP에서는 비교표 화면 조회로 대체
- 실시간(1시간 이내) 수집 — 하루 2회 배치 + 수동 트리거로 충분
- 경쟁사 재무 데이터 자동 수집 — 스펙 비교 항목 중 매출 등은 관리자 수동 입력

#### 3.2 기능 상세

**[경쟁사/제품 등록·관리]**
- 설명: 모니터링 대상 회사와 해당 회사의 제품(솔루션)을 등록·수정·삭제한다
- 동작 방식: 관리자가 회사 정보 입력 → 해당 회사 하위에 제품 등록 → 제품별 비교 항목(스펙) 입력
- 입력: 회사명, 설립일, 사원수, 매출, 웹사이트 URL / 제품명, 제품 컨셉, 출시일, 비교 항목값
- 출력: 등록된 회사·제품 목록, 상세 정보 화면
- 예외 상황:
  - 이미 동일한 회사명이 존재하면 "이미 등록된 회사입니다" 경고 후 기존 데이터 링크 제공
  - 필수 필드(회사명, 제품명) 누락 시 유효성 검사 에러
  - 회사 삭제 시 하위 제품·수집 뉴스도 함께 삭제 (확인 모달 표시)

**[뉴스 자동 수집]**
- 설명: 등록된 경쟁사 키워드 기반으로 네이버 뉴스, 구글 뉴스, 기술 블로그에서 기사를 수집한다
- 동작 방식:
  - 배치: 매일 09:00, 18:00 스케줄러 실행 → 각 회사의 검색 키워드로 뉴스 검색 → 중복 제거 → DB 저장
  - 수동: 관리자가 특정 회사의 [지금 수집] 버튼 클릭 → 즉시 수집 실행
- 입력: 회사별 검색 키워드 목록 (예: "한글과컴퓨터", "한컴", "hwp", "hwpx")
- 출력: 수집된 뉴스 목록 (제목, 본문 요약, 출처, URL, 수집일시, 긍부정)
- 예외 상황:
  - 수집 소스 장애(네이버 API 오류 등) → 해당 소스 실패 로그 기록, 다른 소스는 정상 진행
  - 중복 기사 → URL 기준 중복 제거 (동일 URL은 무시, 유사 제목은 별도 표시)
  - 수집 결과 0건 → "새로운 뉴스가 없습니다" 표시

**[뉴스 AI 분석]**
- 설명: 수집된 뉴스 본문을 LLM으로 분석하여 긍부정 분류(positive/negative/neutral)와 한 줄 요약을 생성한다
- 동작 방식: 뉴스 수집 완료 → 각 기사에 대해 LLM API 호출 → 결과 저장
- 입력: 뉴스 제목 + 본문 (최대 2,000자)
- 출력: 감성(positive/negative/neutral), 한 줄 요약(50자 이내), 관련 키워드 태그
- 예외 상황:
  - LLM API 호출 실패 → 해당 기사는 sentiment=unknown으로 저장, 재시도 큐에 추가
  - 본문 추출 실패 (paywall 등) → 제목만으로 분석, "본문 미확인" 라벨 부착

**[대시보드]**
- 설명: 전체 경쟁 현황을 한눈에 파악하는 메인 화면
- 동작 방식: 페이지 진입 시 최신 데이터 로드
- 입력: 없음 (자동 로드)
- 출력:
  - 등록된 경쟁사 카드 목록 (회사명, 최신 뉴스 수, 최근 감성 추이)
  - 최신 뉴스 피드 (전체 회사 대상, 시간순)
  - 미승인 스펙 변경 알림 배지 (Admin 전용 영역)
- 예외 상황:
  - 등록된 경쟁사 0건 → 빈 상태: "모니터링할 경쟁사를 등록해보세요" + [경쟁사 등록] CTA
  - 수집된 뉴스 0건 → "아직 수집된 뉴스가 없습니다. 첫 수집을 시작해보세요" + [지금 수집] CTA

**[제품 스펙 비교표]**
- 설명: 2~6개 제품을 선택하여 스펙을 열(column)로 나란히 비교한다
- 동작 방식: 사용자가 비교할 제품 체크박스 선택 → [비교하기] 클릭 → 비교표 화면 이동
- 입력: 선택된 제품 ID 목록 (2~6개)
- 출력: 비교 항목(행) × 제품(열) 테이블. 각 셀에 해당 제품의 스펙값 표시
- 비교 항목 관리:
  - **카테고리별 기본 템플릿**: 회사 정보(회사명, 업력, 매출, 사원수), 제품 정보(출시일, 제품 컨셉, 정의), 기술 스펙(제품별 커스텀)
  - **커스텀 항목 추가**: 관리자가 카테고리 내에 비교 항목을 자유롭게 추가/삭제/순서 변경
- 예외 상황:
  - 1개만 선택 시 → "비교하려면 2개 이상 선택하세요" 안내
  - 7개 이상 선택 시 → "최대 6개까지 비교 가능합니다" 안내
  - 특정 항목 값이 없는 제품 → 해당 셀에 "-" 표시

**[뉴스 동향 비교]**
- 설명: 2~4개 회사를 선택하여 최근 뉴스 동향을 나란히 비교한다
- 동작 방식: 사용자가 비교할 회사 선택 → 기간 설정 → 회사별 뉴스 피드가 열(column)로 표시
- 입력: 선택된 회사 ID 목록 (2~4개), 기간 필터 (최근 1주/1개월/3개월)
- 출력: 회사별 뉴스 목록 (시간순), 감성 비율 요약 차트 (긍정/부정/중립 비율)
- 예외 상황:
  - 특정 회사의 뉴스 0건 → 해당 열에 "해당 기간 수집된 뉴스가 없습니다" 표시
  - 5개 이상 선택 시 → "최대 4개 회사까지 비교 가능합니다" 안내

**[스펙 변경 감지·승인]**
- 설명: AI가 수집된 뉴스/블로그에서 제품 스펙 변경 사항을 감지하여 관리자에게 승인 요청한다
- 동작 방식:
  1. 뉴스 분석 단계에서 스펙 관련 변경 사항 감지 (예: "X사, 신제품 v2.0 출시. OCR 정확도 98% 달성")
  2. 감지 결과를 승인 대기열에 추가: 대상 회사, 대상 제품, 변경 항목, 기존값, 신규값, 근거 URL
  3. 관리자가 대기열에서 확인 → [승인] 또는 [거절] (거절 시 사유 입력)
  4. 승인 시 해당 제품 스펙에 자동 반영
- 입력: (System) 뉴스 본문에서 추출한 스펙 변경 정보
- 출력: 승인 대기 목록 화면. 각 항목에 변경 내용 + 근거 링크 + 승인/거절 버튼
- 예외 상황:
  - AI가 잘못 감지한 경우 → 관리자가 거절하면 해당 건은 archived 처리
  - 동일 항목에 대해 여러 건 감지 → 최신 건만 대기열에 표시, 이전 건은 자동 archived
  - 근거 URL이 접근 불가 → "원문 확인 불가" 라벨 표시, 관리자 판단에 맡김

### 4. 사용자 시나리오

**시나리오 1: 관리자가 경쟁사와 제품을 처음 등록한다**

사용자: Admin / 목적: 모니터링 대상 설정

1. 대시보드 접속 → 빈 상태 화면에서 [경쟁사 등록] 클릭
2. 회사 등록 폼에서 회사명 "사이냅소프트", 설립일, 매출, 검색 키워드 ["사이냅소프트", "사이냅 OCR"] 입력
3. [저장] 클릭 → 회사 상세 페이지로 이동
4. [제품 추가] 클릭 → 제품명 "사이냅 도큐애널라이저", 출시일, 제품 컨셉 입력
5. 비교 항목 템플릿(회사 정보, 제품 정보) 자동 생성 → 값 입력
6. 커스텀 항목 [+ 항목 추가] → "지원 문서 포맷", "OCR 정확도" 등 추가 및 값 입력
7. [저장] 클릭 → 등록 완료

검증 기준:
- Given 유효한 회사 정보, When 저장, Then 201 + 회사 목록에 표시
- Given 이미 존재하는 회사명, When 저장, Then 409 + "이미 등록된 회사입니다" 에러
- Given 제품 등록 시 필수 필드(제품명) 누락, When 저장, Then 400 + 유효성 에러


**시나리오 2: 뷰어가 제품 스펙을 비교한다**

사용자: Viewer / 목적: 경쟁 제품 간 스펙 차이 파악

1. 대시보드에서 [제품 비교] 메뉴 클릭
2. 제품 목록에서 "데이터로더", "사이냅 도큐애널라이저", "Doc Parser" 3개 체크
3. [비교하기] 클릭 → 비교표 화면으로 이동
4. 비교표에서 회사명, 업력, 매출, 제품 컨셉, 출시일, 커스텀 스펙 항목이 나란히 표시됨
5. 특정 카테고리(기술 스펙)만 접어서 회사 정보만 볼 수도 있음

검증 기준:
- Given 3개 제품 선택, When 비교하기, Then 3열 비교표 표시
- Given 1개만 선택, When 비교하기, Then "2개 이상 선택하세요" 안내
- Given 특정 항목값 없는 제품, When 비교표 로드, Then 해당 셀 "-" 표시


**시나리오 3: 관리자가 스펙 변경 제안을 승인한다**

사용자: Admin / 목적: AI가 감지한 스펙 변경을 팩트 확인 후 반영

1. 대시보드에서 "미승인 스펙 변경 3건" 알림 배지 확인 → 클릭
2. 승인 대기열 목록에서 첫 번째 항목 확인:
   - 대상: 업스테이지 > Document Parse
   - 변경: "지원 언어" 기존 5개 → 신규 12개
   - 근거: [기사 링크]
3. 근거 링크 클릭하여 원문 확인 → 사실 확인 완료
4. [승인] 클릭 → 해당 제품 스펙에 "지원 언어: 12개" 자동 반영
5. 두 번째 항목: 근거가 불확실 → [거절] 클릭 → 사유 "원문에서 확인 불가" 입력

검증 기준:
- Given 승인 대기 항목 존재, When 승인, Then 제품 스펙 업데이트 + 대기열에서 제거
- Given 승인 대기 항목 존재, When 거절, Then 스펙 미변경 + archived 처리 + 거절 사유 저장
- Given 대기열 0건, When 페이지 접속, Then "검토할 변경 사항이 없습니다" 표시


**시나리오 4 (예외): 뉴스 수집 소스 장애**

사용자: System / 목적: 수집 실패 시 안정적 처리

1. 09:00 배치 수집 시작
2. 네이버 뉴스 API 타임아웃 발생
3. 네이버 소스 실패 로그 기록, 구글 뉴스 + 블로그 수집은 정상 진행
4. 수집 완료 후 관리자에게 "네이버 뉴스 수집 실패 (1/3 소스)" 표시

검증 기준:
- Given 네이버 API 장애, When 배치 수집, Then 나머지 소스 정상 수집 + 실패 로그 기록
- Given 전체 소스 장애, When 배치 수집, Then "전체 수집 실패" 로그 + 대시보드 경고 표시

| 필드명 | 타입 | 제약조건 | 설명 |
|--------|------|---------|------|

### 10. 제약사항 및 가정

**제약사항**:
- 네이버/구글 뉴스 API의 호출 제한(Rate Limit)에 따라 수집 속도 조절 필요
- LLM API 호출 비용이 수집량에 비례하므로 일일 수집 상한 설정 (500건)
- 뉴스 본문 크롤링은 사이트별 robots.txt 정책을 준수하며, paywall 콘텐츠는 제목만 수집
- MVP에서는 한국어 뉴스만 수집 (영문은 P3)

**가정**:
- 사내 네트워크 내에서만 접근하므로 MVP에서 별도 인증 불필요
- 등록 경쟁사 수는 초기 20개 이내, 제품은 100개 이내로 가정
- 관리자(Admin)는 1~3명으로, 스펙 승인 워크플로에 별도 권한 위임 구조 불필요
- LLM 제공자(OpenAI)가 안정적으로 서비스 제공한다고 가정 (장애 시 retry 로직으로 대응)

### 11. 용어 정의

| 용어 | 정의 |
|------|------|
| 경쟁사 (Company) | 모니터링 대상 회사. 자사도 비교 목적으로 등록 가능 |
| 제품 (Product) | 경쟁사 하위의 특정 솔루션 또는 서비스 (예: 데이터로더, Doc Parser) |
| 스펙 (Spec) | 제품의 정량/정성적 사양 정보 (비교 항목 + 값) |
| 비교 항목 (Spec Field) | 스펙 비교표의 행(row)에 해당하는 항목 정의 (예: OCR 정확도, 지원 포맷) |
| 스펙값 (Spec Value) | 특정 제품의 특정 비교 항목에 대한 실제 값 |
| 스펙 변경 요청 (Spec Change Request) | AI가 감지한 스펙 변경 사항. 관리자 승인 전까지 pending 상태 |
| 감성 분석 (Sentiment Analysis) | 뉴스의 논조를 긍정/부정/중립으로 분류하는 AI 분석 |
| 검색 키워드 (Search Keyword) | 회사별로 등록된 뉴스 검색용 키워드 목록 |
| 배치 수집 (Batch Collection) | 스케줄러에 의해 정해진 시각에 자동 실행되는 뉴스 수집 |
| 수동 트리거 (Manual Trigger) | 관리자가 버튼을 눌러 즉시 실행하는 뉴스 수집 |
